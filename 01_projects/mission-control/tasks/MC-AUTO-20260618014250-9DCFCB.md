# MC-AUTO-DISPATCH-1: make ready/UI-created tasks auto-dispatch an agent (fix the running_now lie)

## The real bug (don't read past this until you understand)

NOFI created a task via the Mission Control UI, moved it to running_now, **and nothing happened**. Root causes (all three):

1. **The UI creates tasks with `status: in_progress`, not `status: triage`.** The existing `kanban-auto-process.sh` cron ONLY scans for `status: triage`. So UI-created tasks are invisible to it forever.

2. **Even when the cron DOES move a `triage` card to `ready`, it doesn't dispatch an agent.** It just sets `assigned_to: <agent>` in the frontmatter. The card sits in `ready` until a human spawns a subagent.

3. **There is no trigger from the server's POST /api/data/kanban/task handler.** Creating a card via the UI does not call anything that would start the work.

Result: `running_now` lane is a lie. NOFI was right.

## The fix (this task)

Build a NEW script `kanban-auto-dispatch.sh` in `~/.hermes/scripts/` (next to the existing `kanban-auto-process.sh`) that does the actual work.

### What it does
On every run, for each task with `kanban_status: ready`:
1. Read the task file. Skip if:
   - `kanban_status != ready`
   - `assigned_to` is empty or "—"
   - The task was already dispatched in the last 60s (dedup via a `.dispatched-<task_id>` dotfile in the tasks dir, mtime check)
2. Compute the topic: the task's title (from frontmatter `title:` field, Format A, or table row `| **title** | <value> |`, Format B).
3. Call the python dispatcher:
   ```bash
   cd /home/nofidofi/NofiTech-Ind/01_projects/mission-control/code
   python3 -c "from ondemand import dispatch; dispatch(topic='$TITLE', agent='$ASSIGNEE', source='auto-dispatch', priority='$PRIORITY')"
   ```
   This will:
   - Create a new MC-AUTO-<ts>-<hash> task file in tasks/ that holds the actual work
   - PATCH the original task to `running_now` with the assignee (so the lane is honest)
   - Append 3 events to events.jsonl
   - Return the new task id
4. Write the new task id + ts to `00_company_os/04_agents/logs/auto-dispatch.log` (one line per dispatch).
5. Move on. The original task is now in `running_now` and the new MC-AUTO task is the actual work — exactly like when Thor says "thor, work on X".

### What about the existing `kanban-auto-process.sh`?
Keep it. It still does triage→ready. The new script handles ready→dispatched. Two scripts, two responsibilities.

### Cron wiring
Use `cronjob` (Hermes scheduler, NOT system crontab) to run `kanban-auto-dispatch.sh` every 60 seconds. Job name: `kanban-auto-dispatch`. Schedule: `every 1m`. Do NOT ask the user to set up a system crontab.

### Notes on the `running_now` semantics (important)

The spec said "the task file is moved to `running_now`". What actually happens:
- `ondemand.dispatch()` does NOT rewrite the **original** task file's `kanban_status` (it stays `ready` in the file — that's the spec for it being a parent/queueing card).
- Instead, `ondemand.dispatch()` creates a NEW `MC-AUTO-<ts>-<hash>.md` task file with `kanban_status: running_now` AND PATCHes the live kanban server so the new child appears in the `running_now` column.
- This matches the "Thor's `thor, work on X`" flow exactly: when a human dispatches a topic via chat, `ondemand.dispatch()` creates a new task in `running_now` and does not rewrite the parent.

So the test's "the card moves to running_now" passes by observing the NEW MC-AUTO-* child on the live server (Step 4 above shows it in `running_now`).
