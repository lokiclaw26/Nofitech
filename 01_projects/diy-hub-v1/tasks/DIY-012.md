**Owner:** Forge (build + tests) — Thor self-verifies (read-only) — Argus has no browser tool
**Source:** NOFI 2026-06-20 ~12:45 Dubai
**Defaults (NOFI "go"):** 2C, 3C, 4A, 5A
**Priority:** CRITICAL — user-facing search quality

## Hard rules (DO NOT VIOLATE)
- ❌ NO API keys
- ❌ NO paid APIs
- ❌ NO Google APIs
- ❌ NO Octopart, DigiKey API, Mouser API, Nexar, or any auth-required source
- ✅ FREE public sources only: Wikimedia Commons, Wikidata, Wikipedia, PlatformIO (boards only), Adafruit/SparkFun/Arduino/RPi/Espressif/Wemos/Lolin/ST/TI/Microchip/Bosch Sensortec/NXP/Analog Devices public product pages + PDF datasheets, GitHub public pages (supporting evidence only)

## Why this exists

Current Add Component flow is "search heuristics that lie." It returns:
- ESP32 DevKit V1 → "ESP32-S3 DevKitC-1" (variant error, Adafruit took first result)
- Wemos D1 Mini → "Wemos D1 Mini Pro" (variant mismatch)
- BME280 → BMP280 photos (related but wrong)
- TP4056/LM358/INA219 → PlatformIO pollution (PlatformIO shouldn't show for non-boards)
- Wrong result with image outranks correct result without image
- Short tokens (D1, V1, R3) get dropped or weakened
- Wikimedia Commons image-only results treated as product identity

NOFI wants **reliable recognition, not search.**

## Backend requirements (rules 1-10)

### 1. Query normalization + classification
Classify queries into types:
- `dev_board` — Wemos D1 Mini, ESP32 DevKit V1, Arduino Uno R3, NodeMCU, RPi Pico
- `sensor_module` — BME280, INA219, DHT22, BMP280
- `display_module` — ILI9488, ILI9341, SSD1306
- `touch_module` — XPT2046
- `charger_power_module` — TP4056, BQ25895
- `ic_chip` — LM358, NE555, ATmega328P
- `passive` — resistors, capacitors (skip for now, defer)
- `unknown` — fallback

**Default Q2=C:** hand-curated dict for known parts (~50 entries, you write `tp4056`/`lm358`/`bme280`/`ina219`/`ili9488`/`xpt2046`/`wemos d1 mini`/`esp32 devkit v1`/`arduino uno r3` and similar), heuristic for unknown (token patterns: `*board*`/`*devkit*` → dev_board; `*sensor*`/`*bme*`/`*ina*` → sensor; etc.)

### 2. Preserve short tokens
- Keep tokens < 3 chars IF they contain digits OR are common electronics identifiers
- Recognized short tokens: `D1`, `V1`, `R3`, `S3`, `C3`, `C6`, `A0`, `5V`, `3V3`, `S2`, `R4`, `C6`, `P4`, `Pico`, `Mini`, `Pro`
- Tokenize first, then filter — do NOT drop in the splitter

### 3. Restrict PlatformIO
- USE PlatformIO ONLY when query is dev_board
- DO NOT USE PlatformIO for: TP4056, LM358, INA219, BME280, ILI9488, XPT2046, resistors, capacitors, ICs, sensors, displays, chargers
- Implementation: `if query.component_type == "dev_board": include_platformio(); else: exclude_platformio()`

### 4. Match-level classification (every candidate)
Each candidate gets:
```python
match_level: 'exact' | 'variant' | 'related' | 'image_only' | 'rejected'
component_type: str  # dev_board / sensor / ic / etc.
confidence: float  # 0.0-1.0
confidence_reason: str  # human-readable explanation
source_quality: 'vendor' | 'datasheet' | 'platformio' | 'wikipedia' | 'wikidata' | 'wikimedia_commons' | 'github' | 'unknown'
matched_sources: list[str]
rejected_reason: str | None  # if rejected
```

### 5. Ranking (identity first, image second)
- `exact` match > `variant` match > `related` match > `image_only`
- Within a tier: vendor/datasheet > platformio > wikipedia > wikidata > commons > github
- Identity score (token overlap) > image presence > popularity
- A wrong variant with image MUST NOT outrank an exact match without image
- A generic image-only Commons result MUST NOT outrank a token-exact no-image result

### 6. Variant detection (Q3=C)
- Token-set baseline: compute overlap; missing tokens = penalty, extra tokens = penalty
- Known-variant dict: `{"d1 mini pro": "variant_of d1 mini", "esp32-s3": "variant_of esp32", "esp32-c3": "variant_of esp32", "bmp280": "related_to bme280", "ili9341": "related_to ili9488"}`
- Heavily penalize variant mismatches

### 7. Identity vs image separation
- Phase 1: identity search (vendor, datasheet, wikipedia, wikidata, platformio) — establish canonical identity
- Phase 2: image search (Wikimedia Commons) — fetch images for the identified canonical entity
- Wikimedia Commons should NEVER define identity unless backed by Wikidata/Wikipedia/vendor

### 8. Adafruit/SparkFun scraping (multi-result)
- Inspect at least 5 search results, not just the first
- Rank by exact token match + variant compatibility
- Reject products where important tokens conflict (e.g., "ESP32-S3 DevKitC" rejected for "ESP32 DevKit V1" query because "S3" + "DevKitC" tokens conflict)
- Implement `scrape_vendor_search_multi(query, vendor, max_results=10)` that returns ranked list

### 9. Official-source enrichment
For each candidate, try to fetch:
- Manufacturer/vendor product page URL
- Public datasheet PDF URL (PDF link on vendor page)
- Wikipedia/Wikidata entry (if not already)
- Wikimedia Commons image URL

Cache enrichment results so we don't re-fetch the same URL every search.

### 10. Local verified cache
SQLite table `verified_components` in `diy-hub.db` (default Q4=A):
```sql
CREATE TABLE verified_components (
    id INTEGER PRIMARY KEY,
    query_signature TEXT UNIQUE,  -- normalized query
    canonical_name TEXT,
    manufacturer TEXT,
    model_number TEXT,
    component_type TEXT,
    description TEXT,
    voltage TEXT,
    interfaces TEXT,  -- JSON list
    specs TEXT,  -- JSON object
    datasheet_url TEXT,
    image_url TEXT,
    source_urls TEXT,  -- JSON list
    verified_at TEXT,  -- ISO timestamp
    verified_by TEXT  -- 'thor' | 'forge' | 'argus' | 'nofi'
);
```

On save, NOFI confirms → component goes into verified_components. Future searches: check verified_components FIRST; if hit, return that immediately with high confidence.

## Frontend requirements (rules 1-5)

### 1. No auto-confirm weak single results
- Single result + confidence < 0.85 → show "Needs review" pill (yellow/orange)
- Single result + confidence >= 0.85 → "Exact match" badge (green)
- Multiple results → show all with badges

### 2. Badges
- **Exact match** (green) — confidence >= 0.85
- **Variant** (yellow) — same family, different model
- **Related** (orange) — same domain, different part
- **Image only** (grey) — has image, no real identity match
- **Datasheet-backed** (purple) — has vendor datasheet PDF
- **Vendor-backed** (blue) — has vendor product page
- **From local verified cache** (gold star) — verified_components hit

### 3. Show confidence_reason
Below each result: "Exact model tokens matched: ESP32, DevKit, V1. Image found from Wikimedia. Datasheet not found."

### 4. Edit before save
Edit fields visible BEFORE save:
- name
- manufacturer
- model_number
- category/type
- voltage
- interfaces
- image URL
- datasheet URL

### 5. Reject action
- "Wrong result" button per candidate
- Reject reason text field (optional)
- Server records rejection in `rejected_components` table
- Rejected queries never return that candidate again
