## 3. Surgical code change: `data_orders()` filter in serve.py

The brief section 2 says: *"A simple fix: ALSO exclude any order_id that has a follow-up event with status=cancelled or resolved. Document this fix if you make it."*

I made this fix because the verification spec requires `count=0` and the original filter (status in `pending|in_progress`) cannot honour that without either (a) editing the original 8 events or (b) the filter change. Option (a) was explicitly forbidden ("DO NOT modify the original event"). Option (b) is the only path.

**Change** (in `01_projects/mission-control/code/serve.py`, function `data_orders()`, ~line 665):
- Two-pass scan over `events.jsonl`.
- Pass 1: collect `order_id`s of any `fix_order` with `status in (cancelled, resolved)` into `superseded_ids`.
- Pass 2: emit only `pending|in_progress` orders, skipping any whose `order_id` is in `superseded_ids`.
- Net effect: when an `order_id` has a follow-up cancellation event, the original `status=pending` line is hidden from `GET /api/data/orders`. The original event stays in the log unchanged.

This is the minimal, explicit fix the brief named. It does not change the data model, schema, or any other endpoint.

**Server restart:** required to load the new filter. The brief's "NO server restart needed" clause was conditional on no code change. Brief clause 2 explicitly permits the code change with documentation, so I performed 2 clean restarts (no port-bind issues, idempotent `start-mc.sh`).
