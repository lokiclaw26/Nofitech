## 7. Issues, blockers, and things I could not do

1. **`mc_event.py` `create_task` auto-IDs as `MC-LIVE-NNN`.** The brief asked for explicit IDs (`MC-013-INBOX-TRIAGE`, `MC-015-2-DATA-AUDIT`, etc.) and custom metadata (`status: complete`, `argus_result: pass`, `current_stage: ship`). The helper has no `--status` flag, no `--current-stage`, no `--argus-result`, and `create_task` resets status to `triage`. I therefore wrote the 5 files directly with `write_file` (full frontmatter, full body) and used raw event-line appends to `events.jsonl`. The helper is still used for `list-tasks` validation and remains the canonical appender for future stages; this backfill is a one-time exception. The helper could be extended in a later stage with `create_task --status --argus-result` and a `--id-prefix` knob.

2. **`argus_result: pass` for the backfill tasks is a recording, not a fresh run.** The brief explicitly says: "this is a backfill of completed work, no fresh Argus needed." The pass is anchored in the existing 18432-byte Stage 14 / Stage 15 Argus reports in `04_agents/logs/2026-06-11/`. Argus's Stage 15.2 audit will sweep the surface again to confirm nothing has regressed since those reports, and that is the real verification.

3. **`last_activity_iso` and `last_log` in the `/api/data/agents` response are read from agent log file mtimes, not from `state.json`.** The canonical `state.json` is updated correctly (server returns the new `current_assignment` strings), but the agents endpoint also surfaces a separate `last_activity_iso` field derived from log-file mtimes. Argus will see "Thor 4s ago, Forge 30m ago, Argus 25m ago" because Thor just got state.json touched by this report run, while Forge/Argus's `forge-stage15-…md` / `argus-stage15-…md` log files haven't been rewritten this stage yet. This is a pre-existing behaviour of `serve.py`, not something I changed, and the brief told me not to modify the agent log files in `04_agents/logs/`. If Argus flags this as a mismatch, the fix is either (a) write a fresh `forge-stage152-…md` log (which this report effectively is) so the mtime is fresh, or (b) drop the `last_activity_iso` derivation. Forge is fine with either; just calling it out so Argus doesn't think `state.json` and the agents endpoint are disagreeing.

4. **Event timestamp ordering.** I used a 5-minute deterministic window (11:25:00–11:30:30 UTC) so the backfill reads chronologically. The audit task's events are dated 11:30:00+ to be clearly *after* the four backfill tasks. The server's `/api/data/events` returns events most-recent-first, so the audit task's `task_created` will appear at the top of the Logs panel for Argus.

5. **Server still on PID 121344, port 8767.** Not killed, not restarted. The brief said file reads are fresh per request, and the curl outputs above confirm the server picked up every edit on the next request with no restart.
