# Argus Overbuild Audit — RG-010 (Stage 10)

Argus role: overbuild blocker + existing-prototype safety. v0.1 is a complete 357-line prototype (single `index.html`, fillRect only, no libraries). NOFI's verbatim rules: "Keep the game simple. Keep HTML5 Canvas + JavaScript. No external libraries. No new engine. No procedural generation yet. No inventory system. No complex art system. No sound yet unless NOFI approves later. I want to improve the game step by step, not all at once."

Read before answering: `01_projects/roguelike-v1/code/index.html` (357 lines, read in full), `tasks/RG-010-FUN-VISUAL-UPGRADE-PLAN.md`, `design/technical-values.md`. Per `technical-values.md`: canvas 640×480, TILE=32, grid 20×15, floor cols 1..18 rows 1..13, player HP=5, slime HP=3, attack 1 dmg / 400ms cooldown, slime 500ms greedy Manhattan tick, states `playing|win|lose`, R restarts in win/lose.

## 1. Room Obstacles

In scope. Per `technical-values.md` the room is currently "Inner walls: NONE (one open room)." Adding a small fixed set of stone pillars (e.g. 4–6 hand-placed tiles in cols 1..18, rows 1..13) is **not** procedural generation — it is a static, hand-authored data table, well under NOFI's "keep simple" line. It does not break the win/lose/restart system: slime still dies on hp=0, player still dies on hp=0, R still restarts. The only mechanical touch point is `inBoundsAndFree()` (lines 274–278) — both `movePlayer` and `slimeTick` already route their bounds check through it (well, `movePlayer` inlines its own, but the structure is clean to extend). A new `obstacles` array (or a small `OBSTACLES` constant) of `{col, row}` tuples is the only new data structure, and it is genuinely worth the complexity: it gives the slime AI a reason to use its tie-break order and gives the player cover. **LOC estimate: ~30–50** (obstacle constant, render pass after floor, update `inBoundsAndFree` to reject obstacle tiles, no resetGame change because obstacles are static). This is a NEW mechanic (line-of-sight / pathing) layered on top of existing movement, not a polish pass. It is SAFE for v0.2 and the highest-value-per-LOC candidate on the list.

**Verdict:** ACCEPT — tactical depth from a hand-authored static array, no procedural generation, no engine change, well-contained edit.

## 2. Better Visual Style

In scope, **with a hard guard rail**. NOFI's "no complex art system" rule means: no sprites, no images, no path-heavy SVG, no font loading — stay on `fillRect` and the existing `16px monospace` text. Within that guard, "dungeon floor pattern" = a 2-tone checkerboard or subtle noise in the existing floor loop (lines 82–87), "stone walls" = a darker grey rectangle plus a 1px highlight stroke on the existing wall loop (lines 89–99), "knight player" = the existing 32×32 blue square plus 1–2 small fillRect accents (a grey helmet cap, a yellow visor), "blob slime" = swap the hard square (line 102) for `ctx.beginPath(); ctx.arc(...)` with two magenta shades, "better HUD layout" = the existing HUD text moved into a small dark-bordered box. This is POLISH only — zero new mechanic, zero new data structure. **LOC estimate: ~60–100** depending on how much detail goes into the knight/blob silhouettes. The risk is that "knight" and "blob" become an art project that bloats the render function past 100 LOC and creeps toward the "complex art system" line NOFI banned. Cap it: the knight must remain ≤ 4 fillRect calls, the slime one `arc` plus one fillRect highlight. Visual style is also the FOUNDATION for the other stages — juice (sword slash arc), door (locked vs unlocked colours), potion (red bottle) all reuse this re-skin. Do it first.

**Verdict:** ACCEPT — high-value polish, foundation for v0.2's look-and-feel, provided the knight/blob stay simple fillRect and the HUD layout change is a border not a refactor.

## 3. Combat Feedback / Juice

In scope, but the brief bundles **four** sub-features (slime hit flash, sword slash arc, "-1" damage numbers, screen shake) and that is a scope trap. Per NOFI's "step by step" rule, juice should be one stage and ship as a unit, but the four pieces must be sized to the prototype's existing flash pattern (yellow `attackFlash`, red `playerFlash`, both lines 55 and 57) rather than a new animation system. New data structures: a `damageNumbers` array of `{col, row, value, until}` (push on hit, render with `fillText` and a slight upward `row` offset, expire after ~600ms), and a `screenShake` object `{offsetX, offsetY, until}` (translate the ctx at the top of `render`, restore at the bottom). Both are worth the complexity. The slime hit flash reuses the existing `attackFlash` mechanism extended to the slime tile. The sword slash arc is a `ctx.beginPath(); ctx.arc(...)` from the player toward the facing tile, drawn for the 400ms cooldown — borderline art but cheap if kept to a single arc. **LOC estimate: ~50–90** if all four ship. New mechanic? No — pure polish on the existing Space-attack and contact-damage flows. SAFE for v0.2, but `resetGame()` (line 334) MUST be updated to clear `damageNumbers` and `screenShake`, otherwise the shake can persist across R-restart and damage numbers can leak into the next run. Hard constraint for the builder: no animation loop, no `requestAnimationFrame` — the existing "re-render on every event" pattern must be preserved, so damage numbers expire via a `Date.now() < until` check inside `render()`.

**Verdict:** ACCEPT — essential combat feel, but ship all four sub-features in one stage with a hard "reuse the flash pattern, no animation loop" rule.

## 4. Exit Door Objective

In scope, and this is the most important candidate because it is the only one that **REPLACES** an existing mechanic. Stage 7 currently maps `slime.hp === 0 → state = 'win'` (line 230). This candidate removes that direct mapping and replaces it with: (a) `door` object `{col, row, locked: true}`, (b) slime death flips `door.locked = false` instead of setting `state = 'win'`, (c) `movePlayer` (or a new `checkDoorStep()` called after a successful move) flips `state = 'win'` when the player steps onto an unlocked door tile. The "YOU WIN" overlay (lines 155–163) must change its subtitle from "Slime defeated." to something like "You reached the exit." This is a NEW mechanic (objective-based win condition) and the single biggest semantic change in v0.2. New data structure: a `door` object (or two fields on the slime/level). Worth the complexity? Yes — it converts the prototype from "kill the slime" to "kill the slime AND escape," which is the actual roguelike loop NOFI is heading toward. **LOC estimate: ~30–50** (door object, render branch for locked vs unlocked, win-condition change in `playerAttack` and `movePlayer`, overlay text update, reset restores `door.locked = true`). One red flag: a QA test of the v0.1 win condition ("hit slime 3 times → win overlay") will now FAIL — the test must be rewritten to "hit slime 3 times → door unlocks → step on door → win overlay." This is a test-surface change, not a code-surface risk. SAFE for v0.2, but isolate it as its own stage so the diff is reviewable.

**Verdict:** ACCEPT — the v0.2 win condition, clean isolated state-machine change, but it REPLACES Stage 7's "slime hp=0 = win" mapping and any v0.1 win-condition test must be rewritten.

## 5. One Health Potion

In scope, and the cleanest "step by step" candidate on the list. NOFI's "no inventory system" rule is honored because there is no inventory — one fixed tile, one pickup, one effect, one-shot. New data structure: a `potion` object `{col, row, consumed: false}`. Render: a small fillRect cluster (e.g. dark red bottle silhouette on the floor). Pickup: inside `movePlayer`, after a successful move, if `!potion.consumed && player.col === potion.col && player.row === potion.row`, set `potion.consumed = true` and `player.hp = Math.min(5, player.hp + 1)`. Reset: `potion.consumed = false` in `resetGame()`. **LOC estimate: ~20–35**. New mechanic (item pickup) but trivial — no menus, no carrying, no keybind, no slot, just "walk over, +1 HP, gone." The HP cap at 5 is the one subtle bit: the player can over-heal only if they pick it up at HP=5 and it should be a no-op (cap at 5). SAFE for v0.2.

**Verdict:** ACCEPT — smallest candidate, honors "no inventory," pure one-shot pickup.

- **Stage 15 — One Health Potion** — final new mechanic. Smallest stage, ~20–35 LOC, pickup-only with HP cap at 5.
- **Stage 16 — v0.2 QA Checkpoint** — full regression against WASD/Arrows move, Space attack, R restart, slime 500ms tick, contact damage, win/lose overlays (old AND new door-step trigger), HUD, plus the new v0.2 surfaces (obstacle pathing, juice reset on R, door locked→unlocked, potion pickup + HP cap + consumed-once + reset).

## Overbuild Red Flags

- **Candidate 3 (Juice) is the only real red flag.** The brief lists four sub-features. If the builder treats them as four separate animation systems (timers, requestAnimationFrame, easing functions) it will balloon past 100 LOC and break the "event-driven re-render" pattern that v0.1 is built on. Mitigation: enforce a single rule — every juice effect expires via `Date.now() < until` checked inside the existing `render()` function, no new loop. The sword slash arc is the most art-risk piece: keep it to one `ctx.arc` over the 400ms cooldown, no rotation, no particle tail.
- **Candidate 4 (Door) is a SEMANTIC red flag, not a code red flag.** It replaces the win condition, so any QA, smoke test, or "I just want to see the win overlay" demo flow that was written for v0.1 must be updated. No code is at risk; the test surface is. Flag this for the QA stage explicitly.
- **Candidate 2 (Visual Style) has a soft red flag.** "Knight" and "blob" can drift toward a "complex art system" if the builder starts layering > 4 fillRect calls per sprite or starts loading any image. Cap it. The player is still fundamentally a coloured square; the slime is still fundamentally a coloured shape. No sprites, no paths with more than two commands.
- **No procedural generation in any candidate.** All three "new data" candidates (obstacles, door, potion) are fixed hand-authored constants, not random placement. This is correct and must stay correct.
- **No sound in any candidate.** NOFI explicitly gated sound on separate approval. None of the five introduces AudioContext or `<audio>`. ✅
- **No external libraries in any candidate.** ✅

## Existing-Prototype Safety

Risk-by-mechanic for v0.1's working features:

- **WASD/Arrows move (`movePlayer`, lines 179–193):** Safe from all 5 candidates. The keydown listener (lines 236–268) is unchanged by all 5. Obstacles extend the bounds check; door adds a post-move win check; potion adds a post-move pickup check — all additive, all AFTER the bounds check. None of them rewire the keydown handler.
- **Space attack (`playerAttack`, lines 208–233):** Safe. Juice adds a render-time arc, not a logic change. The 400ms cooldown guard stays. Door changes what happens on the killing blow (`slime.hp === 0` no longer sets state, it unlocks the door) but the damage + flash flow is unchanged.
- **R restart (`resetGame`, lines 334–347):** **Medium risk** — must be updated. Juice adds `damageNumbers` and `screenShake` state that must be cleared. Potion adds `potion.consumed` that must be reset. Door adds `door.locked` that must be reset. Obstacles are static so no reset needed. Visual style does not affect reset. This is a checklist item for Stages 13, 14, 15, not a blocker.
- **Slime 500ms tick (`slimeTick`, lines 284–305, `setInterval(..., 500)` line 353):** Safe. Obstacles extend `inBoundsAndFree`; the 500ms cadence is untouched. Door, potion, juice, and visual style do not touch the tick.
- **Contact damage (`applyContactDamage`, lines 311–329):** Safe from all 5 candidates. The Manhattan ≤ 1 check and the 1-HP decrement are unchanged.
- **Win/lose overlays (lines 155–173):** **One targeted risk** — the "YOU WIN" subtitle text changes for the door objective (Stage 14). "YOU DIED" overlay and the freeze-on-non-playing-state behaviour are unchanged. The overlays must still be drawn LAST so they sit on top of juice and visual style.
- **HUD (`Player HP`, `Slime HP`, `State:`, lines 127–143):** **Soft risk** for visual style. The brief says "better HUD layout" — the current HUD uses `fillText` at fixed pixel offsets (8, 280, `W-130`). A "better" layout must keep the same three pieces of information in non-overlapping positions and must not collide with the centered "DUNGEON SPARK" title (line 146) or the end-game overlays (which draw their own font sizes at 240 and 270). The Stage 11 builder should sketch the new HUD offsets before coding.
- **`render()` ordering (lines 76–174):** The current order is floor → walls → slime → attackFlash → player → playerFlash → HUD → state text → end overlay. Adding obstacles goes between walls and slime. Adding the door goes after slime (and changes colour). Adding the potion goes after the floor / before the player (so the player walks over it). Adding the sword slash arc goes with `attackFlash` (line 109). Adding the slime hit flash goes between slime and the attack flash, OR reuses the attackFlash slot. Adding screen shake wraps the whole `render()` body in `ctx.save(); ctx.translate(offX, offY); ...; ctx.restore();`. Adding damage numbers goes between the HUD and the end overlay so they sit above the scene but under the win/lose dim. All additive, none of them re-order existing layers.
- **Tile constants (`COL_MIN/COL_MAX/ROW_MIN/ROW_MAX`, lines 73–74):** Unchanged by all 5. Obstacles are stored as a separate set, not by mutating these constants.

**Summary:** No candidate breaks a v0.1 working feature. The Stage 11 builder must sketch new HUD offsets before coding. The Stages 13, 14, and 15 builders must each add a line to `resetGame()`. The Stage 14 builder must update the "YOU WIN" subtitle text. Other than those three checklist items, the diff is additive.
