## Stage Order Conflict — Resolution

Forge and Argus agreed on all 5 candidates being ACCEPT. They disagreed on the first stage.

- **Forge (Obstacles first):** "Every later stage (#2 visuals, #4 door, #5 potion) needs to know the room can contain obstacles so it can place its own tiles (door, potion) in a non-blocked location. Doing obstacles last would force door/potion placement to retroactively avoid pillars."
- **Argus (Visual Style first):** "Visual style is the FOUNDATION that juice (Stage 13), door (Stage 14) and potion (Stage 15) all build on."

**Thor's resolution: Forge's order wins.** The dependency that matters is **physical placement**, not visual layering. The door and the potion need to know where pillars are *before they pick their constant tile position*, otherwise the builder will have to refactor the constant mid-stage. Visual style can be done over an obstacle layout OR over an empty room — it works either way — but door and potion cannot pick a non-conflicting position until obstacles are known.

**If** NOFI prefers Argus's order (Visual Style first, Obstacles second), the trade-off is: door and potion must be re-positioned in Stage 14/15 if obstacles are added at their chosen tile. The total LOC is similar; the review surface is just split differently. Either order is safe; Forge's is cleaner.

**Recommended final order (this report):**
- **Stage 11 — Room Obstacles** ← NOFI approves this next
- **Stage 12 — Better Visual Style**
- **Stage 13 — Combat Feedback / Juice**
- **Stage 14 — Exit Door Objective** (replaces Stage 7's win condition)

## Why Each Stage Comes In This Position

- **11 Obstacles (first):** Static layout. Every later stage references it. Lowest risk per LOC. Sets the spatial model.
- **12 Visual Style (second):** Largest LOC stage (+50-80). Pure cosmetic, no logic. Done after obstacles so the knight/blob sprites are drawn over a room that already has pillars, not redrawn twice.
- **13 Juice (third):** Polish on top of the new look. Four sub-features (slime flash, slash arc, -1 numbers, shake) in one stage. Hard rule: no `requestAnimationFrame` — must reuse the existing `Date.now() < until` event-driven re-render pattern. `resetGame` extended to clear new state.
- **14 Exit Door (fourth):** The semantic risk candidate. Replaces Stage 7's `if (slime.hp === 0 && state === 'playing') state = 'win';` with a "kill slime → door unlocks → step on door → win" flow. Comes late so the room is structurally + visually + feel-complete before the win flow is rewritten. Updates "YOU WIN" subtitle text and `resetGame`.
- **15 Potion (fifth):** Smallest + most isolated. New `potion {col, row, consumed}` state. Pickup-only, +1 HP capped at 5. `resetGame` extended.
- **16 QA (last):** Full regression against WASD/Arrows, Space attack, R restart, slime 500ms tick, contact damage, win/lose overlays (new door flow), HUD, plus the new v0.2 surfaces (obstacle pathing, juice reset on R, door locked→unlocked, potion pickup + HP cap + consumed-once + reset). No code — playtest + checklist.

## Hard Constraints Inherited From V0.1 (Locked)

- HTML5 Canvas + vanilla JS only
- No external libraries (no Three.js, no Phaser, no Pixi, no jQuery)
- No new engine
- No procedural generation (obstacles, door, potion are all hand-authored constants)
- No inventory system (potion is a one-shot tile pickup, no carrying, no keybind, no slot)
- No complex art system (knight ≤ 4 fillRect, slime ≤ 1 arc + 1 fillRect, no sprites, no images, no font loading — keep `16px monospace`)
- No sound (NOFI explicit; would require separate approval)
- No new game modes
- No multiplayer
- No save/load
- No mobile export

## Risks Flagged For Future Stages

- **Stage 11:** must update all THREE bounds-check sites (`movePlayer` lines 184-185, `playerAttack` line 220, `inBoundsAndFree` lines 275-276) when adding `obstacles[]`, or entities will walk through pillars.
- **Stage 12:** "knight" and "blob" can drift toward "complex art system" if builder layers > 4 fillRect per sprite or loads any image. Cap it.
- **Stage 13:** screen shake is the trickiest sub-feature. The end-game overlay + HUD must be drawn OUTSIDE the shake transform or the "YOU WIN" headline shakes too — looks broken.
- **Stage 14:** three specific failure modes — (1) forget to remove the line-230 win trigger, (2) door tile surrounded by obstacles / unreachable, (3) player standing on door tile when slime dies (door opens mid-tick, need to check the door step *after* the player has moved).
- **Stage 15:** potion must not spawn on (2,2) player spawn, (17,12) slime spawn, future door, or any obstacle. HP cap must be `Math.min(5, hp+1)` or the player can exceed 5 HP.

## Scope-Check Question For NOFI

The 5 candidates are not all-or-nothing. If NOFI wants a SHORTER v0.2, the recommended cut order is:
1. **Drop Potion (Stage 15) first** — most isolated, least strategic value, easiest to add later.
2. **Drop Door (Stage 14) second** — most risk. If dropped, the win condition stays as "kill slime = win" (v0.1 behavior) and v0.2 is pure polish.
3. **Keep Obstacles (11) + Visual Style (12) + Juice (13) always** — these three are the most "fun and visual" per NOFI's brief.
4. **QA (16) is always required** — the new mechanic(s) need regression verification.

Resulting shortened v0.2: Stage 11 + 12 + 13 + 16. ~3 build stages + 1 QA, ~+110-170 LOC. Same "fun and visual" core, half the work.
