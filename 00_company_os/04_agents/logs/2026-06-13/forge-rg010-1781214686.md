# Forge Feasibility Report — RG-010 (Stage 10)

Source reviewed: `01_projects/roguelike-v1/code/index.html` (357 lines, single IIFE, vanilla Canvas, no libs).
Key landmarks in the current code:
- State objects (lines 49-65): `player {col,row,facing,hp}`, `slime {col,row,hp}`, `lastAttackTime`, `attackFlash`, `playerFlash`, `state`.
- Constants (lines 68-74): `TILE=32`, `COLS=20`, `ROWS=15`, `COL_MIN=1`, `COL_MAX=18`, `ROW_MIN=1`, `ROW_MAX=13`.
- `render()` (lines 76-174): one long painter, draw order = clear → floor → walls → slime → attackFlash → player → playerFlash → HUD → overlay.
- `movePlayer()` (lines 179-193): bounds-checks `nextCol`/`nextRow` against `COL_MIN/MAX/ROW_MIN/MAX` only.
- `playerAttack()` (lines 208-233): swing, deal 1 dmg, set `attackFlash`, and the **win trigger** at line 230: `if (slime.hp === 0 && state === 'playing') state = 'win';`.
- `inBoundsAndFree()` (lines 274-278): same edge-only check, used by slime AI.
- `applyContactDamage()` (lines 311-329): **lose trigger** at line 326: `if (player.hp === 0 && state === 'playing') state = 'lose';`.
- `resetGame()` (lines 334-347): restores player/slime positions, HP, cooldowns, flashes, state.

## 1. Room Obstacles

This is the cleanest "extend, don't change" candidate. The bounds-check pattern (`if (c < COL_MIN || c > COL_MAX || r < ROW_MIN || r > ROW_MAX)`) is duplicated in **three** call sites: `movePlayer` (lines 184-185), `playerAttack` target check (line 220), and `inBoundsAndFree` (lines 275-276). All three need to consult a single new `isBlocked(c, r)` helper that combines the edge check with an `obstacles[]` lookup. The render function (lines 76-174) gains a new pass between the wall ring (line 99) and the slime (line 102) to draw 4-6 hardcoded obstacle tiles as darker gray rects with a small inner highlight (still `fillRect`-only, no images). The `obstacles` array is a module-level constant alongside the existing `TILE`/`COLS` constants (lines 67-74). Spawn validation is required: `player` at (2,2) and `slime` at (17,12) must not overlap any obstacle; if they would, either the obstacle list or the spawn must shift. `resetGame` (lines 334-347) is unchanged (obstacles are static). **Effort: +20-30 LOC. Architecture: pure extension. Data: `obstacles[]` constant array of `{col,row}`. State: none (static layout). Win condition: untouched. Regression risk: MEDIUM** — forgetting to update *any one* of the three bounds-check sites would let the player or slime walk through a pillar, or let a swing hit through one. Slime AI's "no candidates → wait" branch (line 296) already handles the case where pillars trap the slime, so no AI rewrite is needed. **Dependency: should come FIRST** — every later stage (#2 visual style, #4 door, #5 potion) needs to know the room can contain obstacles so it can place its own tiles (door, potion) in a non-blocked location. Doing obstacles last would force door/potion placement to retroactively avoid pillars.

## 2. Better Visual Style

Lives entirely inside `render()` (lines 76-174) and the CSS in `<style>` (lines 6-32). Floor fill at line 82 (`#333`) becomes a 2-shade checker pattern using two nested loops and alternating `fillStyle`. Wall fill at line 91 (`#000`) gets a 2-pixel inner border in a lighter gray to suggest stone. Player fill at line 115 (`blue`) becomes a 3-rect knight: body (blue), helm (lighter blue), sword tip (white) in the `player.facing` direction. Slime fill at line 102 (`rgb(180,40,180)`) becomes a blob: a central rect plus two offset "lobes" using overlapping fillRects (no arcs, no paths — staying within the `fillRect` rule from `technical-values.md`). HUD at lines 127-149 is reorganized: HP shown as 5 red rects ("hearts") instead of `Player HP: 5` text, slime HP similar. The title text at line 146 (`DUNGEON SPARK`) gets a slight drop-shadow by drawing it twice in offset positions. **Effort: +50-80 LOC. Architecture: extension, but a small refactor is warranted** — `render()` will exceed ~200 lines if everything stays inline, so extracting `drawFloor()`, `drawWalls()`, `drawPlayer()`, `drawSlime()`, `drawHUD()` helpers is a natural side-effect. The refactor itself is low-risk if draw order is preserved. **Data: none. State: none. Win condition: untouched. Regression risk: LOW-MEDIUM** — the main risk is draw-order mistakes during the helper extraction (e.g., drawing the player *before* the slime hides the player). A second minor risk: if the obstacle pass from #1 is in place, the new floor pattern must skip obstacle tiles (or be drawn first and then the obstacle overlays it). **Dependency: should come AFTER #1** so the obstacles are already in the layout when the visual style is applied — otherwise the new floor pattern is drawn, then obstacles are added later as plain rects, and the visual style has to be re-applied to them. This stage is the largest single LOC delta in v0.2.

## 3. Combat Feedback / Juice

Four sub-features, each a small addition. **(a) Slime hit flash**: new `slimeFlash = {col:-1, row:-1, until:0}` state (alongside `attackFlash`/`playerFlash` at lines 55-57). Set in `playerAttack` at line 224 where `attackFlash` is already set; drawn in `render()` between the slime (line 103) and `attackFlash` (line 109) so the flash sits on top of the slime but under the swing tile. **(b) Sword slash arc**: replace the flat yellow square at lines 109-112 with a small triangular slash — three `fillRect` calls forming a wedge in the `player.facing` direction. No new state, just a render change. **(c) Damage number "-1"**: new `damageNumbers[]` array of `{col, row, value, until}`. Pushed in `playerAttack` line 224-225 area when a hit lands. Drawn in `render()` after the slime and after the slash (so the number floats above the action), filtered each frame to drop expired entries. Uses `ctx.fillText` with a 200-400ms lifetime. **(d) Screen shake on player damage**: new `shake = {until:0, magnitude:0}` state. Set in `applyContactDamage` at line 327 (right where `playerFlash` is set). Applied in `render()` by wrapping the **scene draw only** in `ctx.save()` / `ctx.translate(randomOffset, randomOffset)` / `ctx.restore()` — the overlay (lines 155-173) and HUD text (lines 127-149) must be drawn *outside* the shake transform so the "YOU WIN" headline and the restart hint stay still. **Effort: +40-60 LOC. Architecture: extension. Data: `slimeFlash`, `damageNumbers[]`. State: `shake`. Win condition: untouched. Regression risk: MEDIUM** — the four sub-features are independent but the screen shake is the trickiest. The most common bug: wrapping the entire `render()` in save/translate/restore shakes the end-game overlay too, which looks broken. A second risk: damage numbers can drift into the HUD area at the top of the canvas. A third: the slime flash and the attack-flash yellow square can stack and obscure the swing. The shake should also be reset in `resetGame` (line 334) to prevent a stale shake carrying into a new run. **Dependency: should come AFTER #2** so the polish lands on the new knight/slime sprites, not on the old plain squares (otherwise we polish gray, then re-polish the new look). Putting this BEFORE #4 is fine because the door's own open animation is independent of these four juice effects — none of them target the door, so no integration is required at this stage.

## 4. Exit Door Objective

**This is the only candidate that REPLACES an existing behavior rather than extending it.** The current win trigger at line 230 in `playerAttack`:
```js
if (slime.hp === 0 && state === 'playing') state = 'win';
```
must be **removed** (or rewritten to just set `doorOpen = true` without flipping state). The new win trigger lives in `movePlayer` after the bounds-check passes: `if (doorOpen && player.col === DOOR_COL && player.row === DOOR_ROW) state = 'win';`. New state: `doorOpen` (boolean, default `false`) and a constant door position (e.g., `(1, 13)` — bottom-left corner of the floor, away from player spawn at (2,2) and slime spawn at (17,12)). Door rendering: new pass in `render()` after walls (line 99) and before the slime, drawing a distinct rect — closed door is a red/brown square, open door is a green/yellow square with a small inner glow (2-3 nested `fillRect`s). The end-game overlay text at lines 155-173 must be updated: the current "Slime defeated. Press R to restart." is wrong for the new flow — it should read something like "You escaped! Press R to restart." The lose overlay (lines 164-173) is unchanged. `resetGame` (lines 334-347) must reset `doorOpen = false`. **Effort: +25-40 LOC. Architecture: CHANGES the win condition.** This is the highest-risk candidate in v0.2. **Data: door position constant. State: `doorOpen`. Win condition: REPLACED.** **Regression risk: MEDIUM-HIGH** — three specific failure modes to guard against. (1) Forgetting to remove the line-230 win trigger means killing the slime still ends the game immediately and the door is never reached. (2) If the door tile is unreachable (e.g., surrounded by obstacles added in #1, or placed in a corner the slime can box the player into), the game becomes unwinnable. The door position must be validated against `obstacles[]` from #1. (3) The player can be standing on the door tile when the slime dies — in that case `doorOpen` flips to true mid-tick and the next `movePlayer` (or even the current one's post-move state) must check. The cleanest implementation checks the door *after* the player has moved and after all state guards. **Dependency: should come AFTER #1, #2, and #3** — needs obstacles in place (so door placement is validated), needs the visual style (so the door looks like a door, not a random rect), and ideally comes after juice (so the slime's death is a satisfying moment *before* the player walks to the door). Putting door early means the "open door" animation gets a later pass when juice is added, which is wasted work.

## 5. One Health Potion

The smallest and most isolated candidate. New state object `potion = {col, row, consumed: false}` declared near the other state at lines 49-65. Position is a constant (e.g., `(9, 6)` — roughly center of the 18x13 floor, away from spawns, the future door, and any future obstacles; must be re-validated if #1 adds obstacles at the chosen spot). Render pass: a small green/white rect (one `fillRect` of 16x16 centered in the 32x32 tile) drawn after the floor (line 87) and before entities, only if `!potion.consumed`. Pickup check: inside `movePlayer` after the bounds-check passes and before the position update, check `if (!potion.consumed && player.col === potion.col && player.row === potion.row) { player.hp = Math.min(5, player.hp + 1); potion.consumed = true; }`. The `Math.min(5, ...)` enforces a cap — currently no HP cap exists in the code (HP only ever decrements), so this is a *new* invariant. `resetGame` (lines 334-347) sets `potion.consumed = false` and restores `player.hp = 5`. **Effort: +15-20 LOC. Architecture: pure extension. Data: `potion {col, row, consumed}`. State: `potion.consumed`. Win condition: untouched.** Potion can be picked up at any time, including after the slime is dead and the door is open — a player at full HP can still walk over it (it's a no-op because `consumed` is already true, or they get a wasted +1 if not at 5 HP yet). **Regression risk: LOW** — three small things to watch. (1) Potion must not spawn on the player spawn (2,2), the slime spawn (17,12), the door (chosen in #4), or any obstacle (from #1). (2) The HP cap must be enforced or the player can exceed 5 HP, which would let them survive more than the designed 5 hits. (3) Picking up the potion while the slime is adjacent should not trigger or block `applyContactDamage` — but since pickup is in `movePlayer` and damage is in `applyContactDamage` (called at the end of `slimeTick`), they never collide on the same frame. **Dependency: should come LAST among gameplay changes** — it's the smallest, most isolated, and doesn't touch the win condition, the door, or the combat loop. It can absorb a small amount of risk because if it breaks, nothing else does.

## Recommended Stage Order

1. **Stage 11 — Room Obstacles** — smallest structural change, sets the spatial model that every later stage (#2 visuals, #4 door, #5 potion) needs in order to place its own tiles safely. No win-condition impact, low risk per LOC added.
2. **Stage 12 — Better Visual Style** — the largest single stage in v0.2 (~+50-80 LOC), but it's pure cosmetic with no logic change. Doing it after #1 means the new floor pattern and knight/slime sprites are drawn over a room that already has obstacles, so we don't redraw obstacles twice.
3. **Stage 13 — Combat Feedback / Juice** — polish layer (slime flash, slash arc, damage numbers, screen shake). Comes after the visual style so the new knight/slime look is what gets polished, not the old plain squares. Screen shake is the trickiest piece and benefits from a dedicated stage so regressions are easy to bisect.
4. **Stage 14 — Exit Door Objective** — **REPLACES** Stage 7's win condition (line 230 must be removed or rewritten). By this stage the room is structurally complete (#1), visually polished (#2), and the slime fight feels good (#3), so the new "kill slime → walk to door → win" flow is tested end-to-end with the full v0.2 treatment. Coming late means the highest-risk change has the most context around it.
5. **Stage 15 — One Health Potion** — smallest change (+15-20 LOC), isolated pickup, doesn't touch the win condition or the door. Safe last because if it breaks, nothing else regresses; if everything else breaks, this stage is small enough to bisect against.
6. **Stage 16 — v0.2 QA Checkpoint** — full playthrough with the new win flow (kill slime, door opens, walk to door, win). Verify: movement still bounds-checks, attack still works, contact damage still works, restart still resets all v0.2 state (`doorOpen=false`, `potion.consumed=false`, new `damageNumbers[]` cleared, `shake` cleared, `slimeFlash` cleared). No code in this stage — playtest + regression checklist.
