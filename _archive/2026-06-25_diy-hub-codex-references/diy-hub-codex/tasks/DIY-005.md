# DIY-005 — Stage 5: Real Live Component Lookup

## Why
The current Add Component flow reads a pre-set list of ~16 components from `mock_data.py`. "Wemos D1 Mini" is not in the list, so the parser returns a synthetic-fallback candidate with no image and no real data. **This is wrong for a real DIY inventory tool.** NOFI (operator) has now approved live internet lookup with explicit constraints.

## NOFI approval (verbatim)
> "Real Live Component Lookup. NOFI approves free public internet lookup.
> Allow remote HTTP requests only to public no-key sources: manufacturer pages, GitHub docs, Wikimedia/Wikipedia, PlatformIO board data, Arduino/ESPHome docs, and public datasheet PDFs.
> Do not use paid APIs, API keys, login-required sources, Google Custom Search, Octopart, or purchasing actions."

## Goal
Replace mock-only lookup with real live lookup. When the operator types any component name (including one not in any pre-set list), the system searches free public sources, returns real variants with real images and real specs, and asks the operator to confirm before saving anything.

## Scope

### IN-SCOPE
- Live HTTP calls to 5 allowed sources (all free, all no-key, all no-login)
- Variant candidate picker (model option cards) when multiple matches found
- Real image download to `data/images/` ONLY after ADD TO DATABASE confirmed
- Confirmation popup showing: real image, name, model, category, voltage, interfaces, key specs, tags, datasheet/source links, confidence score
- Mock catalog retained as **offline fallback only**, clearly labeled "mock fallback" in UI
- "No reliable live result found" message + manual entry path when all live sources fail
- New DB columns: `wikidata_id`, `commons_filename`, `source_url`, `manufacturer`, `release_year`, `confidence`, `datasheet_url`

### OUT-OF-SCOPE
- Touching Mission Control, RGV1, or any project under `01_projects/` other than `diy-hub-v1`
- Inventing specs (no fake/guessed values, no fallback to mock data unless the operator explicitly clicks the offline fallback toggle)
- Saving unconfirmed search results (must always go through ADD TO DATABASE button)
- Any code that requires authentication, API keys, or paid plans
- Google Custom Search, Octopart, distributor scraping for pricing, or any purchasing action

## Live source priority order

For every `query` from the operator, run these in order and merge results:

1. **PlatformIO board database** (`https://platformio.org/boards/...`) — highest priority for MCU boards, returns structured MCU/freq/flash/voltage
2. **Wikidata** (`wbsearchentities` + `wbgetentities`) — manufacturer, release year, description, official website, image filename
3. **Wikimedia Commons** (`list=search` on srnamespace=6) — real photos
4. **Wikipedia REST summary** — description text
5. **GitHub public repos** (search API + raw readme fetch) — last resort, slow, not always structured

User-Agent: `DIY-Hub/1.0 (https://github.com/nofitech) +mailto:ops@nofitech.local` (polite, identifies project)

## Required operator flow (per NOFI brief)

1. Operator types component name + model (one input, free text)
2. Backend searches live online (all 5 sources, in parallel with 5s timeout each, merged)
3. If multiple variants are found, show model option cards (frontend picker)
4. Operator selects one variant
5. Backend fetches detailed info for the selected variant (re-fetches the relevant source URLs, extracts specs)
6. Confirmation popup shows:
   - Real image (if found) — never a colored block, never a fake placeholder unless labeled as fallback
   - Name
   - Model / variant
   - Category (MCU board, sensor, regulator, etc.)
   - Voltage
   - Interfaces (I2C, SPI, UART, etc.)
   - Key specs
   - Tags
   - Datasheet/source links (clickable)
   - Confidence score (0.0 - 1.0, based on how many sources agreed)
7. Operator clicks ADD TO DATABASE or CANCEL
8. Only on ADD TO DATABASE:
   - Save to SQLite
   - Download real image to `data/images/<slug>.<ext>` (skip download if no real image)
   - Store local image path in DB

## Fallback rules (locked)

- If all 5 live sources fail or return 0 candidates:
  - Show `"No reliable live result found"` message
  - Show a button "Enter manually" (operator can type name/category/voltage/etc by hand)
  - Show a separate button "Try offline mock fallback" (clearly labeled, uses the 16-component `mock_data.py` list — operator must explicitly choose this)
- If 1-2 sources fail but others succeed:
  - Proceed with whatever was found
  - Confidence score reflects partial coverage
  - Show in the popup: "Live lookup partial: PlatformIO and Wikidata unavailable, used Wikimedia only"
- If a candidate has NO real image:
  - Show a gray "No real image found" placeholder with source attribution "Source: live lookup · partial coverage"
  - Never a colored block, never a generated SVG, never a fake placeholder
- If download fails after ADD TO DATABASE:
  - Save the record anyway with `image_path = null`
  - Log warning to backend log
  - Tell operator: "Saved to DB, but image download failed. You can re-attempt from Inventory page."
